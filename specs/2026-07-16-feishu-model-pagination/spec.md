# 飞书模型卡片分页

## 背景

OpenCode 当前可返回超过 20 个模型，而飞书 `/model` 卡片只展示前 20 个模型按钮，后续模型无法通过卡片访问。

## 目标

- `/model` 首次打开模型卡片时显示第 1 页。
- 每页最多展示 20 个模型按钮。
- 模型总数超过一页时显示上一页和下一页按钮。
- 点击分页按钮时更新原模型卡片，不额外发送新的模型卡片消息。
- 所有可用模型均可通过翻页访问。
- 保留当前模型、Recent、配置模型、其余 provider 模型的优先顺序。

## Requirement IDs

| ID | Requirement |
| --- | --- |
| REQ-001 | `/model` 卡片按每页最多 20 个模型分页，并显示当前页和总页数。 |
| REQ-002 | 分页按钮使用现有命令回流协议，动作格式为 `cmd:/model --page <页码>`。 |
| REQ-003 | 点击分页按钮更新原卡片；第一页没有上一页，末页没有下一页。 |
| REQ-004 | 模型完整排序为当前模型、Recent、配置模型、其余 provider 模型，模型在完整序列中只出现一次。 |
| REQ-005 | 页码缺失、非数字、小于 1 或超过总页数时安全归一化到有效页。 |
| REQ-006 | 分页导航不受普通命令去重影响，允许用户在相邻页面之间往返。 |
| REQ-007 | 直接输入 `/model <provider>/<model_id>` 的模型切换行为保持不变；卡片失败时保留纯文本 fallback。 |

## 设计

### 卡片渲染

`renderModelListCard(models, options)` 接收新增的 `options.page`：

- 页码采用 1-based 表示，默认第 1 页。
- 先生成去重后的完整有序模型序列，再按每页 20 个模型切片。
- 卡片标题显示模型总数，正文显示 `第 X / Y 页`。
- 导航按钮使用单个 action 行承载上一页和下一页。
- 当前模型和 Recent 使用 primary 按钮，其余模型使用 default 按钮。

### 命令处理

`MessageDispatcher._cmdModel()` 将 `/model --page <页码>` 识别为列表分页请求，而不是模型切换请求：

- 重新从当前 session agent driver 获取模型目录，避免缓存过期。
- 将页码、当前模型和分页动作来源的卡片消息 ID传给 `sendModelList`。
- `/model` 无参数仍打开第 1 页。
- 分页动作跳过普通命令去重；重复 patch 同一页是幂等操作。

### 飞书 API 挂载

`sendModelList` 在收到 `options.updateMessageId` 时调用 `patchCard` 更新原卡片，否则沿用 `replyCard` 创建卡片。

## 边界

- 不新增模型搜索。
- 不持久化页码。
- 不改变 OpenCode 模型目录接口。
- 不把导航按钮计入每页 20 个模型按钮限制。
- 不伪造 OpenCode 未提供的 Recent 数据。

## 验收

- 53 个模型生成 3 页，页模型数量依次为 20、20、13。
- 第 1 页仅有下一页，第二页同时有上一页和下一页，末页仅有上一页。
- 点击下一页后调用 `patchCard` 更新触发动作的卡片消息。
- 从第二页返回第一页不会被命令去重阻止。
- 每个模型恰好出现在一个页面中，三页按钮并集覆盖全部 53 个模型。
- 模型切换、纯文本 fallback 和现有卡片 routeKey 回流测试继续通过。
