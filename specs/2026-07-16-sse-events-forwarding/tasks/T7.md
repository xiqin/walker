## 实现步骤

- [ ] 先把旧 permission-question 测试夹具改为原生 `question.asked`、`question.replied`、`question.rejected` 和 protocol v4 runtime。
- [ ] 所有前置任务完成后读取其实际实现和测试导出，按最终接口建立跨层 fixture，不复制前序任务的内部实现。
- [ ] 覆盖一次多问题 asked 产生多张卡片，分别提交单选、多选和自定义答案，最终只生成一次顺序正确的 `question_reply` delivery。
- [ ] 验证插件 accepted 后调用原生 question SDK，且不调用 `promptAsync`；成功 final 驱动飞书全部卡片进入已处理状态。
- [ ] 覆盖本地 TUI replied/rejected 抢先、重复卡片回调和并发最后一题；细粒度路由、TTL、容量及全部非法表单由 T5/T6 单元测试负责。
- [ ] 覆盖 accepted 前明确失败后重试成功，以及 accepted 后 final 丢失进入 processed_unknown 且无法重试。
- [ ] 覆盖 protocol v3 runtime 的安全降级；卡片发送次数、feishu_unavailable 和终态反馈细节由 T6 单元测试负责。
- [ ] 构造父 prompt 已 leased 且 session busy 的嵌套场景：控制 poll 只取 question_reply，完成后父 prompt 继续 heartbeat/final，队列中的其他 prompt/clear 顺序不变。
- [ ] 保留独立 permission 请求与 `replyPermission`、普通消息 prompt、clear delivery 的回归断言。
- [ ] 运行目标集成测试和完整项目测试，修复仅由本功能引入的失败；不得回滚工作树中的其他变更。
