# Task 3: 插件转发原生事件并执行控制 delivery

## 描述

升级生成的 OpenCode 插件，使其转发原生 question 事件，并在 session busy 或父 prompt 尚未完成时独立执行 protocol v4 question reply 控制 delivery。

## 依赖

- 前置 task：T1、T2
- 外部依赖：已安装的 `@opencode-ai/sdk` question API

## 涉及文件

| 操作 | 文件路径 | 说明 |
| ---- | -------- | ---- |
| 修改 | `src/opencode-hook/plugin-template.js` | 升级插件版本、事件映射、poll 策略和控制 delivery 执行器 |
| 修改 | `test/opencode-hook-installer.test.js` | 动态执行生成插件并验证 SDK 与 Bridge 交互 |

## 复杂度

high，原因：生成代码需同时处理事件映射、父/控制并发状态、accepted 重试、heartbeat/final 和 SDK 调用边界。

## 实现步骤

- [ ] 先增加失败测试，将生成插件版本递增并注册 `bridgeProtocolVersion: 4`。
- [ ] 前置任务完成后读取其实际 AgentEvent 与 Bridge 实现，确认字段、错误对象和 accepted 响应与固定接口一致。
- [ ] 监听 `question.asked`、`question.replied`、`question.rejected`，分别发送 T1 定义的事件类型和完整字段；无效事件只记录错误，不污染其他事件。
- [ ] 将父 prompt/clear delivery 与 question reply 控制 delivery分开追踪；控制状态至少区分 `awaitingAccepted`、`executingSdk`、`finalizing`，分别保存 ID、heartbeat timer、SDK 调用标记和 final 状态。
- [ ] 每次 tick 优先处理已有 `awaitingAccepted` 控制 delivery 的 accepted 重试，再决定是否 poll；session idle 且无活跃父 delivery时以 `['prompt', 'clear', 'question_reply']` poll，session busy 或父 prompt 活跃时仅以 `['question_reply']` poll。
- [ ] 已有任意活跃控制状态时不得重新 poll 控制队列或建立第二份控制状态；tick 防重入机制不得阻止后续 accepted 重试。
- [ ] 校验 question reply 的 `requestID` 和 `answers`，校验失败时上报 `sdkInvoked: false` 的 final，且不得调用 SDK。
- [ ] 只有 accepted 成功且未返回 `expired: true` 后才调用 `api.client.question.reply({ requestID, answers })`；accepted 临时失败时保留 `awaitingAccepted` 状态并由后续 tick 重试，不调用 SDK或重新 poll。
- [ ] accepted 返回 `expired: true` 时清理本地控制状态，不调用 SDK，也不发送与 Bridge tombstone 冲突的第二个 final。
- [ ] SDK 调用前在本地标记 `sdkInvoked: true`；成功和失败 final 均携带稳定 `code`、`deliveryPhase`、`sdkInvoked`、`safeToRetry`。
- [ ] 确认 question reply 从不调用 `promptAsync`，控制 final 不会清除或覆盖父 prompt final。
- [ ] 回归普通 prompt、clear、permission 事件和插件安装更新行为。
