# Task 6: 实现 question 状态机与 Dispatcher 路由

## 描述

新建 QuestionHandler 管理多问题卡片、答案收集、整组提交、并发终态和有界保留，并将 MessageDispatcher 从旧 permission-question 复用迁移到独立原生事件路由。

## 依赖

- 前置 task：T1、T4、T5
- 外部依赖：飞书消息与卡片 API

## 涉及文件

| 操作 | 文件路径 | 说明 |
| ---- | -------- | ---- |
| 创建 | `src/dispatch/question-handler.js` | 原生 question 请求状态机和飞书交互逻辑 |
| 修改 | `src/dispatch/message-dispatcher.js` | 路由三类事件和 `/answer` 命令到 QuestionHandler |
| 修改 | `src/dispatch/permission-handler.js` | 移除原生 question 复用职责，保留纯 permission 行为 |
| 创建 | `test/question-handler.test.js` | 状态机和卡片交互单元测试 |
| 修改 | `test/message-dispatcher.test.js` | 验证事件与命令路由 |
| 修改 | `test/permission-handler.test.js` | 验证 permission 链路不回归 |

## 复杂度

high，原因：包含多题状态、异步卡片发送、原子提交权、路由隔离、非幂等失败分类、终态竞态和 TTL/容量淘汰。

## 实现步骤

- [ ] 先创建失败测试，固定 QuestionHandler 的外部职责：处理 asked/replied/rejected 和 `/answer --form|--retry`，内部键使用 `agentRef + requestID`。
- [ ] 前置任务完成后读取实际 AgentEvent、Driver 和卡片构建器实现，确认接口与 `plan.md` 固定契约一致后再接线。
- [ ] 建立 `sending_cards`、`collecting`、`submitting`、`replied`、`rejected`、`processed_unknown`、`feishu_unavailable` 请求状态，以及等长的 answers/cards/cardStates/cardAttempts。
- [ ] asked 时逐题构建 T5 卡片；每次调用底层 `feishuApi.replyCard` 前递增 `cardAttempts`，单题最多调用 2 次，不使用默认尝试 3 次的通用发送封装。
- [ ] 任一卡片两次均失败或问题无选项且禁止 custom 时进入 `feishu_unavailable`，停止继续收集，并 patch 已发送卡片提示本地 TUI 回答。
- [ ] 解析 `question_selected` 与 `question_custom`：验证 option 序号、单选互斥、多选按原 options 顺序并追加 trim 后的自定义文本；每题只保存首个合法答案。
- [ ] 使用回调中的 questionKey、Walker session ID、routeKey、messageId 校验原请求；Walker session 必须仍映射到保存的 agentRef，不得使用点击时当前焦点 agent。
- [ ] 最后一题完成时原子切换为 `submitting`，只调用一次 `driver.replyQuestion(agentRef, requestID, answers)`；重复和并发回调只 patch“正在处理”或当前终态。
- [ ] MessageDispatcher 对 `/answer` 跳过通用 MessageDedup，让同一卡片的重复回调和多次合法重试始终进入 QuestionHandler；其他命令继续使用原去重策略。
- [ ] 仅对 `safeToRetry: true && sdkInvoked: false` 回到 collecting 并显示统一重试按钮；protocol 不支持转为 feishu_unavailable；request not found、SDK 已调用或未知错误转为 processed_unknown。
- [ ] replied/rejected 事件可从非明确终态抢先结束请求；processed_unknown 可被迟到明确终态校正；replied 与 rejected 之间最先观察到的明确终态不可反转。
- [ ] `submitAnswers()` 的迟到 resolve/reject 必须重新检查当前状态：明确终态已由事件形成时不得反转、重复 patch 或改成 processed_unknown；迟到结果仅记录诊断信息。
- [ ] 所有状态反馈优先 patch 原卡片；单次 patch 失败后向原 chatId 发送一次文本，反馈失败不改变业务状态。
- [ ] 对四类终态执行 24 小时 TTL 和合计 1000 条容量淘汰；状态不存在、过期或被淘汰时 patch“请求已过期”。
- [ ] MessageDispatcher 注册独立事件路由并将 `/answer` 交给 QuestionHandler；删除旧 questionReplyStates 和 permission 分支中的原生 question 处理。
- [ ] 按 T5 固定的 `/answer <questionKey> --form|--retry <walkerSessionId>` 参数契约路由命令，不重新定义或兼容旧手工 answer 协议。
- [ ] PermissionHandler 只保留 permission 请求及回复职责，现有 `replyPermission` 调用路径不变。
