# Task 2: 升级 TUI Bridge protocol v4

## 描述

升级 Bridge 的事件白名单、question reply 入队协议、按类型 poll、控制 delivery 租约和 accepted 幂等语义，为 busy session 中的原生回复提供可靠通道。

## 复杂度

high，原因：涉及队列保序、两类并发 delivery、租约、幂等 accepted、超时 tombstone 和错误可重试性判定。

## 实现步骤

- [ ] 先更新失败测试，将原生回复接口固定为 `replyQuestion(sessionRef, requestID, answers)`，载荷固定为 `{ requestID, answers }`。
- [ ] 在入队前检查目标 runtime 的 `bridgeProtocolVersion >= 4`；不满足时返回 `QUESTION_REPLY_UNSUPPORTED`、`deliveryPhase: 'preflight'`、`sdkInvoked: false`、`safeToRetry: false`。
- [ ] 扩展事件规范化白名单，接受 T1 固定的三类 question AgentEvent，同时保持 permission 等事件行为。
- [ ] 为 poll 增加可选 `acceptedTypes`：省略时继续 FIFO；提供时取第一条匹配项并保持其他项顺序。
- [ ] 同一 session 可排队多个 `question_reply`，但存在 dequeued 或 leased 控制 delivery 时，控制型 poll 返回 `null` 且不取出第二条；当前控制 delivery 完成、超时或放弃后再按原队列顺序取下一条。
- [ ] poll 取出 `question_reply` 时记录 `dequeuedAt` 并立即启动 accepted timer，允许它与一个父 prompt delivery 并存；timer 使用 runtime stale 窗口，默认 10 秒。
- [ ] accepted timer 到期时主动删除 pending、释放 session 的控制占用、写入 `accepted_timeout` tombstone，并以 `deliveryPhase: 'queued'`、`sdkInvoked: false`、`safeToRetry: true` 的结构化错误拒绝 `replyQuestion()` Promise；该收敛不得依赖后续 poll、report 或 cleanup 请求。
- [ ] accepted 首次上报将 queued 转为 leased并清除 accepted timer；同 ID、runtime、session 的 leased 重放返回 `duplicate: true` 且不重置租约；超时后的迟到 accepted 返回 `expired: true`。
- [ ] accepted、final、close、dispose 和其他结束 pending 的路径必须清理 accepted timer及控制占用，且不得覆盖父 prompt pending 状态。
- [ ] Bridge 所有 question reply 失败均 reject 一个保留 `message` 且带 `code`、`deliveryPhase`、`sdkInvoked`、`safeToRetry` 属性的 `Error`；final 从插件错误载荷复制这些字段，不得降级成纯消息。
- [ ] 固定错误分类：protocol 门禁为 `preflight/false/false`，accepted 超时为 `queued/false/true`，accepted 后租约丢失按 `leased/true/false` 保守处理；只有插件明确证明未调用 SDK 时才允许 `sdkInvoked: false`。
- [ ] 回归 prompt、clear、旧版省略 `acceptedTypes` 的 poll、普通租约和 final tombstone。
