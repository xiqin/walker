# Task 2: OpencodeTuiBridge 新增 replyQuestion delivery

## 描述

在 `OpencodeTuiBridge` 新增 `replyQuestion(sessionRef, questionId, answer)` 方法，投递 `type='question_reply'` 的 delivery 到 runtime queue，复用现有 `pending` Map 和 lease/heartbeat/final 确认机制。runtime 通过 poll 拉取后，经 accepted→heartbeat→final 状态流转完成回复回传。

## 依赖

- 前置 task：无
- 外部依赖：无

## 涉及文件

| 操作 | 文件路径 | 说明 |
|------|----------|------|
| 修改 | `src/opencode-tui-bridge/bridge.js` | 新增 `DELIVERY_TYPE_QUESTION_REPLY` 常量、`replyQuestion` 方法、`normalizeEvents` 不变（question_reply 是 delivery 不是 event） |
| 修改 | `test/opencode-tui-bridge.test.js` | 新增 replyQuestion 相关测试 |

## 复杂度

high，原因：需复用现有 prompt 的 lease/heartbeat/final 状态机，但要投递不同 type 的 delivery；需处理 runtime 不在线/stale/session 变更的边界；需保证 cancel/dispose/close 时清理 question_reply pending。

## 实现步骤

- [ ] 步骤 1：在 `bridge.js` 顶部新增常量 `const DELIVERY_TYPE_QUESTION_REPLY = 'question_reply';`（与 `DELIVERY_TYPE_PROMPT`、`DELIVERY_TYPE_CLEAR` 并列）。
- [ ] 步骤 2：新增 `replyQuestion(sessionRef, questionId, answer)` 方法，复用 `prompt` 方法的结构：
  - `_validateRef(sessionRef)` 校验 transport/runtimeId/opencodeSessionId。
  - 检查 runtime 存在、非 stale、currentSessionId 匹配（与 prompt 一致）。
  - 生成 `deliveryId = createId('del_')`。
  - 构造 delivery：`{ deliveryId, type: DELIVERY_TYPE_QUESTION_REPLY, sessionId: ref.opencodeSessionId, questionId, answer }`。
  - 创建 pendingEntry（state='queued'，resolve/reject/timer/leaseStartedAt/cancelReason），push 到 `runtime.queue`。
  - 返回 Promise（resolve 时无返回值，`resolve()` 即 `resolve(undefined)`，与 prompt 的 `resolve(events)` 区分：replyQuestion 不需要返回 events 给调用方；reject 时抛错）。
- [ ] 步骤 3：确认 `_handleAccepted`、`_handleHeartbeat`、`_handleFinal`、`_loseLease`、`cancel`、`dispose`、`close` 对 `pending` Map 的操作自动覆盖 question_reply（因为这些方法按 deliveryId 操作 pending，不区分 type），无需额外修改。
  - 验证：`_handleFinal` 的 `pending.resolve(events)` 对 replyQuestion 场景语义正确 — runtime final 上报可能不含 events，`resolve([])` 或 `resolve(undefined)` 均可，调用方（`OpencodeDriver.replyPermission` → `tuiBridge.replyQuestion`）不消费 resolve 值，只关心 Promise fulfilled 状态。
  - 验证：`_handleFinal` 的 `pending.reject(new Error(errorMessage(data.error)))` 对 replyQuestion 场景正确 — final 上报 error 时 reject，触发 T7 `_cmdAnswer` 的 catch 回滚 `questionReplyStates` 为 pending（spec §3.2.1 失败条目、§6 异常表）。
  - 验证：`_loseLease` 的 `pending.reject(new Error('TUI_RUNTIME_DISCONNECTED'))` 对 replyQuestion 场景正确 — lease 超时 reject，触发 T7 回滚。
- [ ] 步骤 4：确认 `normalizeEvents` 不需改动（question_reply 是 delivery，不是 Agent 事件类型，不经过 normalizeEvents）。
- [ ] 步骤 5：在 `reportEvents` 中确认 `data.deliveryId` 存在时会走 `_handleAccepted/_handleHeartbeat/_handleFinal` 分支，question_reply delivery 的 final 会上报 `deliveryState: 'final'` 并 resolve replyQuestion 的 Promise。
