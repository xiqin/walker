# Task 7: Dispatcher + PermissionHandler question 分发与回复全链路

## 描述

在 `MessageDispatcher` 新增 `questionReplyStates` Map（3 态机 pending/submitting/replied，失败回滚 pending，含 inputMode/required）、`_cmdAnswer` 方法、handleCommand 的 answer 分支、`_handlePermissionEvent` 按 `data.type==='question'` 分发；同时在 `PermissionHandler` 新增 `handleQuestion` 和 `handleQuestionReplied` 方法，用 `buildQuestionCard`/`buildQuestionRepliedCard` 渲染。dispatcher 与 handler 紧耦合在同一 task，避免中间态不可运行。

## 实现步骤

### Dispatcher 部分

- [ ] 步骤 1：在 `MessageDispatcher` 构造函数中新增 `this.questionReplyStates = new Map();`（与 `permissionCardIds` 并列）。
- [ ] 步骤 2：在 `handleCommand` 的 handlers 表（第 279-300 行）新增：
  ```js
  answer: () => this._withRouteTouch(cmd.routeKey, () => this._cmdAnswer(cmd)),
  ```
- [ ] 步骤 3：新增 `_cmdAnswer(cmd)` 方法，逻辑：
  1. 从 `cmd.args[0]` 取 `questionId`；缺失时回复 `'用法: /answer <questionId> <value> | /answer <questionId> --form'`，返回 `{ error: 'missing_args' }`。
  2. 判断 `cmd.args[1] === '--form'`：
     - 是：从 `cmd.formValue.question_answer` 提取 answer（string 或 string[]）；`formValue` 缺失或 `question_answer` 缺失时回复 `'表单值缺失'`，返回 `{ error: 'missing_form_value' }`。
     - 否：从 `cmd.args.slice(1).join(' ')` 提取 answer（confirm/single_select）。
  3. 从 `questionReplyStates[questionId]` 反查 `required` 字段（在 `_handlePermissionEvent` 分发时存入，见步骤 4）；无记录时默认 `required = true`。校验 `answer` 非空（`required !== false` 时空字符串/空数组拒绝），回复飞书提示 `'必须提供非空回答'`，返回 `{ error: 'empty_answer' }`。
  4. 校验 `questionReplyStates[questionId]` 的 `state`：
     - `replied`：回复 `'问题已回复: ' + questionId`，返回 `{ duplicate: true, state: 'replied' }`。
     - `submitting`：回复 `'问题正在处理中: ' + questionId`，返回 `{ duplicate: true, state: 'submitting' }`。
     - `pending`/无记录：允许提交（失败已回滚为 pending，不存在 failed 终态）。
  5. 取 `current = sessionService.getCurrent(cmd.routeKey)`；无 session 时回复 `'No session bound to this conversation.'`，返回 `{ noSession: true }`。
  6. 取 `driver = driverRegistry.get(current.agent)`；无 driver 或 `driver.replyPermission` 不存在时回复 `'当前 agent 不支持权限回复'`，返回 `{ error: 'driver_not_supported' }`。
   7. 取 `existing = questionReplyStates.get(questionId)`；设置 `questionReplyStates[questionId] = { state: 'submitting', answer, repliedAt: null, inputMode: existing?.inputMode, required: existing?.required }`（保留原 inputMode/required）。
   8. 调用 `driver.replyPermission(current.agentRef, questionId, answer, false)`：
     - 成功：`questionReplyStates[questionId] = { state: 'replied', answer, repliedAt: Date.now(), inputMode: existing?.inputMode, required: existing?.required }`；回复飞书 `'已回答问题 ' + questionId`；返回 `{ replied: questionId, answer }`。
     - **卡片 patch 策略区分 transport**（避免双重 patch 竞态，spec §3.2.2 兜底关系）：
       - HTTP transport（`current.agentRef.transport !== 'tui-bridge'`）：`_cmdAnswer` 直接 patch 卡片（用 `permissionCardIds.get(questionId)` 取 cardId，调用 `buildQuestionRepliedCard(questionId, answer)`），因为 HTTP transport 下 opencode 不一定推送 `permission_replied` SSE 事件，`_cmdAnswer` 的 patch 是主要路径。
       - TUI bridge transport（`current.agentRef.transport === 'tui-bridge'`）：`_cmdAnswer` **不 patch**，等 runtime 上报 `permission_replied` 事件后由 `handleQuestionReplied` patch（spec §3.2.2 步骤 4-5）。若 runtime 未上报 `permission_replied`，则卡片不更新（与 spec §3.2.2 最后一段"也可自行 patch"的兜底语义一致，但为避免竞态优先等事件驱动）。
     - 失败（含 HTTP 回传失败、TUI bridge replyQuestion reject 如 lease 超时/runtime 断连/final error）：**回滚为 pending 允许重试**（spec 规则 8、§6 异常表），即 `questionReplyStates[questionId] = { state: 'pending', answer: null, repliedAt: null, inputMode: existing?.inputMode, required: existing?.required }`；回复 `'问题回复失败: ' + questionId`（TUI bridge runtime 不在线时回复 `'OpenCode TUI runtime 未连接'`，见 spec §6）；记录 warn；返回 `{ error: 'reply_failed' }`。
     注：spec §4.2 的 `failed` 是中间态语义，最终回滚到 `pending`；本实现不保留 `failed` 终态，直接回滚 `pending` 与 spec 规则 8 一致。
- [ ] 步骤 4：修改 `_handlePermissionEvent`（第 1152-1154 行），按 `data.type` 分发并预存 inputMode/required 到 questionReplyStates：
  ```js
  _handlePermissionEvent(session, chatId, agentEvent) {
    const dataType = agentEvent.data && agentEvent.data.type;
    if (dataType === 'question') {
      const questionId = agentEvent.data.id;
      const meta = agentEvent.data.metadata || {};
      this.questionReplyStates.set(questionId, {
        state: 'pending', answer: null, repliedAt: null,
        inputMode: meta.inputMode, required: meta.required !== false,
      });
      return this.permissionHandler.handleQuestion(session, chatId, agentEvent);
    }
    return this.permissionHandler.handle(session, chatId, agentEvent);
  }
  ```
- [ ] 步骤 5：修改 `_handlePermissionRepliedEvent`（第 1156-1158 行），按 questionReplyStates 是否有记录分发（只有 question 分支才在 `_handlePermissionEvent` 中写入 questionReplyStates）：
  ```js
  _handlePermissionRepliedEvent(session, chatId, agentEvent) {
    const permissionId = agentEvent.data && agentEvent.data.permissionId;
    const state = this.questionReplyStates.get(permissionId);
    if (state) {
      return this.permissionHandler.handleQuestionReplied(session, chatId, agentEvent);
    }
    return this.permissionHandler.handleReplied(session, chatId, agentEvent);
  }
  ```

## 关键接口契约（跨 task 一致性）

- `questionReplyStates` 结构：`Map<string, { state: 'pending'|'submitting'|'replied', answer: string|string[]|null, repliedAt: number|null, inputMode: string|null, required: boolean }>`
  - 在 `_handlePermissionEvent` 分发时初始化为 `{ state:'pending', answer:null, repliedAt:null, inputMode, required }`。
  - `_cmdAnswer` 和 `handleQuestionReplied` 都从中反查 `inputMode` 和 `required`。
  - 失败时回滚为 `pending`（spec 规则 8、§4.2 `failed → pending`），不保留 `failed` 终态。
- `buildQuestionRepliedCard(questionId, answer)` — 2 参数，T4 定义，T7 调用时传 2 参数（inputMode 在 replied 卡片无用途）。
- `permissionCardIds` 继续用于 id→cardId 映射，question 和 permission 共用（spec §4.2）。
