# Task 4: MessageDispatcher 异步错误与状态收敛

## 描述

为消息和命令处理补齐错误边界，确保异步飞书回复失败被捕获，并让 prompt 完成后的状态写入尊重 session 终态。

## 依赖

- 前置 task：T2, T3
- 外部依赖：Node.js `node:test`

## 涉及文件

| 操作 | 文件路径 | 说明 |
|------|----------|------|
| 修改 | `src/dispatch/message-dispatcher.js` | await 或捕获飞书 API Promise；统一命令错误边界；prompt 完成后重新读取 session 状态再决定是否写入 idle/error |
| 修改 | `test/message-dispatcher.test.js` | 增加异步飞书失败、命令失败、prompt 与 stop/delete 竞争测试 |

## 复杂度

high，原因：dispatcher 同时协调 session、driver 和飞书反馈，需避免把外部发送失败扩散为主流程崩溃。

## 实现步骤

- [ ] 步骤 1：新增测试，`sendUnboundGuide`、`addReaction`、`sendErrorCard`、`replyText` 返回 rejected Promise 时不产生未捕获 rejection。
- [ ] 步骤 2：新增测试，`/new` 的 `ensureReady()` 或 `createSession()` reject 时发送错误卡片并返回错误结果。
- [ ] 步骤 3：新增测试，`/use` 无参数或不存在 session 时通过错误卡片反馈，不把异常交给平台层吞掉。
- [ ] 步骤 4：新增测试，prompt 运行期间 session 被 stop/delete，prompt resolve 或 reject 后最终状态仍为 stopped/deleted。
- [ ] 步骤 5：实现安全发送辅助逻辑，统一捕获飞书 API 异步失败并记录日志。
- [ ] 步骤 6：为 `handleCommand()` 增加统一 try/catch，失败时发送错误卡片或文本反馈。
- [ ] 步骤 7：在 prompt 完成和错误路径写状态前重新读取 session，若已是 stopped/deleted 则跳过回写。
