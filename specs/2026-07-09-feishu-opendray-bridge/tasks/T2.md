# Task 2: 路由与 Walker session 核心

## 描述

实现 `routeKey -> walkerSessionId -> agentRef` 的核心绑定、Walker session 生命周期、状态持久化和纯业务命令能力。

## 依赖

- 前置 task：T1
- 外部依赖：无

## 涉及文件

| 操作 | 文件路径 | 说明 |
|------|----------|------|
| 创建 | `src/core/route-key.js` | 生成 thread/user/channel 三种飞书 routeKey |
| 创建 | `src/core/session-store.js` | 封装 `sessions.json` 和 `routes.json` 的读写结构 |
| 创建 | `src/core/session-service.js` | 创建、绑定、解绑、查询、停止、删除 Walker session |
| 创建 | `test/route-key.test.js` | 覆盖 routeKey 生成规则 |
| 创建 | `test/session-service.test.js` | 覆盖绑定、恢复、状态切换和删除 |

## 复杂度

medium，原因：核心状态机需要精确，但不接触外部网络或子进程。

## 实现步骤

- [ ] 步骤 1：实现 `buildRouteKey(message, mode)`，thread 模式按 `rootId -> parentId -> messageId` 选择根消息，user 模式按 `chatId/openId`，channel 模式按 `chatId`。
- [ ] 步骤 2：实现 `SessionStore`，数据形状与 spec 的 `sessions.json`、`routes.json` 一致，重启后可重载。
- [ ] 步骤 3：实现 `SessionService.createSession({route, agent, title, runtime, cwd, agentRef})`，生成 `wks_` ID 并自动绑定当前 routeKey。
- [ ] 步骤 4：实现 `bindRoute`、`unbindRoute`、`getCurrent`、`listSessions`、`markRunning`、`markIdle`、`markError`、`stopSession`、`deleteSession`。
- [ ] 步骤 5：保证每个 routeKey 同时最多绑定一个 Walker session，一个 Walker session 可包含多个 routeKey。
- [ ] 步骤 6：删除 session 时把状态标记为 `deleted`，并清理所有指向该 session 的 route binding。
