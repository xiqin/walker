# Task 1: Bridge clear 协议与汇合状态机

## 描述

扩展 TUI bridge 和 OpenCode driver，使 Walker 能安全投递 clear 控制请求，并通过 `controlDeliveryId` 关联的暂存注册与 control result 两阶段提交，原子更新 runtime 和 route 焦点。

## 依赖

- 前置 task：无
- 外部依赖：无

## 涉及文件

| 操作 | 文件路径 | 说明 |
|------|----------|------|
| 修改 | `src/opencode-tui-bridge/bridge.js` | 增加 clear delivery、pending 汇合、注册关联、超时和清理逻辑 |
| 修改 | `src/drivers/opencode-driver.js` | 增加仅针对 `tui-bridge` 的 `clearSession()` 委托 |
| 修改 | `test/opencode-tui-bridge.test.js` | 覆盖两种事件顺序、继承、拒绝和超时路径 |
| 修改 | `test/opencode-driver.test.js` | 验证 TUI clear 委托且不访问 HTTP 或打开终端 |

## 复杂度

high，原因：需要处理 control result 与关联 register 任意顺序到达的竞态，隔离同时发生的普通 session 切换，并保证失败时 runtime 和 route 状态不被部分提交。

## 实现步骤

- [ ] 为现有 prompt delivery 增加 `type: 'prompt'`，保持 poll API 结构不变。
- [ ] 新增 `clearSession(sessionRef)`，复用 runtime 在线、心跳和当前 session 一致性校验，投递 `{ deliveryId, type: 'clear', sessionId }`。
- [ ] 投递前检查同一 runtime 是否已有 clear pending；存在时立即拒绝，确保每个 TUI runtime 最多一个在途 clear。
- [ ] 为 clear pending 保存旧 Walker session、原 route、旧模型、新 OpenCode session ID、暂存 cwd/版本、register/control 完成标记、timer 和 Promise 回调。
- [ ] 扩展 `reportEvents()`：识别 `control.type === 'clear'`，校验 runtime、旧 session 和新 ID；error 立即失败，成功结果等待对应 register。
- [ ] 扩展 `register()`：带 `controlDeliveryId` 时必须命中对应 clear pending，并校验 runtime；仅记录 OpenCode session ID、cwd 和版本，不提前创建持久化 Walker session，不更新 `runtime.currentSessionId`、不按 cwd 自动选 route、不设置焦点、不触发普通 enrollment 回调。未知或过期关联 ID 直接拒绝，不回退为普通注册。
- [ ] 普通无关联 register 保持现有行为；clear 自身以外的手工 session 创建或切换不得满足 clear pending。
- [ ] 增加单一完成函数，仅在 control 与关联 register 的 `deliveryId + runtimeId + newSessionId` 完全一致时，创建或复用 Walker session、提交 runtime 当前 session、加入原 route、继承旧模型、设置焦点、触发 enrollment 回调、清 timer 并 resolve；支持两种到达顺序。
- [ ] 在超时、`cancel()`、`dispose()` 和 `close()` 中拒绝并清理相关 clear pending；迟到 control 不提交，迟到关联 register 被拒绝且不得改变焦点。
- [ ] 在 `OpencodeDriver.clearSession()` 中拒绝非 TUI ref，并直接委托 bridge，禁止调用 HTTP create 或 `_openTerminalForSession()`。
