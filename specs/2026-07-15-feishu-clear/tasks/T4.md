## 实现步骤

- [ ] 构造当前焦点为 `tui-bridge` 的真实 Walker session，并设置模型、route 和 cwd。
- [ ] 从 dispatcher 发起 `/clear`，轮询并断言 delivery 为 `type='clear'` 且指向旧 OpenCode session。
- [ ] 分别覆盖 control 先到和关联 register 先到的完整链路，确认命令只在匹配 `controlDeliveryId + runtimeId + newSessionId` 的两者完成后返回成功。
- [ ] 在汇合前断言 runtime 当前 session 和 route 旧焦点均不变；汇合后断言新旧 Walker session 同属原 route、新 session 为焦点、模型与 cwd 继承、旧 session 可通过 `setFocus()` 恢复。
- [ ] 注入无关普通 register 或错误关联 ID，断言其不能完成 clear 或提前改变原 route 焦点。
- [ ] 并发发起同 runtime 的第二个 clear，断言其在投递前失败且只创建一个新 OpenCode session。
- [ ] 断言链路不访问独立 OpenCode HTTP/SSE、不调用 driver `createSession()`，并保留现有 prompt 双向测试通过。
- [ ] 增加失败集成场景：插件 error、关联 register/control 上报失败、超时或 runtime stale 时命令返回错误，Walker 旧焦点不变；验证插件导航后的最佳努力回滚及回滚失败提示。
- [ ] 增加运行中场景：`/clear` 在 route lock 外立即提示先 `/cancel`，当前 prompt 后续完成时也不得自动执行 clear。
- [ ] 在 README 命令表加入 `/clear`；说明仅适用于已连接且空闲的 OpenCode TUI、运行中先 `/cancel`、旧会话可用 `/list` 和 `/use` 恢复。
- [ ] 运行完整 `npm test` 并修复本需求引入的回归。
