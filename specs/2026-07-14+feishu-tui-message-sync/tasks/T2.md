# Task 2: 修复 TUI 回复 watcher 恢复后的即时推送

## 描述

验证并完成 `OpencodeSessionWatcher` 在飞书 prompt 导致 suspend/resume 后的轮询恢复，确保继续调用 watch 注册时的原始 handlers，并立即把已完成的 TUI assistant 消息推送给 dispatcher。

## 依赖

- 前置 task：无
- 外部依赖：无

## 涉及文件

| 操作 | 文件路径 | 说明 |
|------|----------|------|
| 修改 | `src/drivers/opencode-session-watcher.js` | 恢复轮询时正确复用原始 handlers |
| 修改 | `test/opencode-driver.test.js` | 覆盖 suspend/resume 后的 TUI 回复投递 |

## 复杂度

medium，原因：涉及 SSE 与轮询的共享 watcher 生命周期、消息游标和异步定时器，必须避免重复投递和测试资源泄漏。

## 实现步骤

- [ ] 审阅工作区已有 watcher/driver 差异，区分本问题修复与其他重构。
- [ ] 使用回归测试对 HEAD 旧实现建立失败信号：watch 初始化后 suspend，新增已完成 assistant 消息，再 resume，应收到 TEXT 和 DONE 事件。
- [ ] 确认 `_resumePolling` 把 watcher 注册时的 handlers 原样传给 `_startMessagePolling`，不额外嵌套 `onEvent`。
- [ ] 使用可配置短轮询间隔和条件等待稳定测试，不依赖固定长 sleep。
- [ ] 验证游标只推进到已处理消息，pending assistant 消息完成后仍可投递，stopWatch 清理定时器。
