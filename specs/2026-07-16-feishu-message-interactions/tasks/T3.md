# Task 3: dispatcher 命令集成、模型 footer 与去重修正

## 描述

把新卡片能力接入命令处理链路，使 `/model`、`/help` 优先发送交互卡片，普通 Agent 回复追加模型 footer，并修正卡片命令去重 key。

## 依赖

- 前置 task：T1、T2
- 外部依赖：现有 `feishuApiTarget` 注入方式、`MessageDispatcher._callFeishu()` fallback 行为、message dedup 语义

## 涉及文件

| 操作 | 文件路径 | 说明 |
|------|----------|------|
| 修改 | `src/dispatch/message-dispatcher.js` | `/model` 无参数改用当前 agent driver 的 `listModels()`；`/help` 优先卡片；追加 footer；修正命令去重。 |
| 修改 | `test/message-dispatcher.test.js` | 增加 dispatcher 集成、footer 和去重测试。 |

## 复杂度

high，原因：集成点集中在 dispatcher，需要同时保持命令兼容、多 agent 约束、回复发送边界和去重语义。

## 实现步骤

- [ ] 步骤 1：调整 `_cmdModel(cmd)` 无参数分支：必须先获取当前会话，再使用该会话的 agent driver 调用 `listModels()`；不再固定 `driverRegistry.get('opencode')`。
- [ ] 步骤 2：当当前 driver 没有 `listModels()` 或返回失败时，返回“不支持模型列表”的用户提示，不回退到其它 agent。
- [ ] 步骤 3：`/model` 无参数优先调用 `sendModelList`；若平台未挂载该能力，则保留现有纯文本列表 fallback。
- [ ] 步骤 4：`_cmdHelp(cmd)` 优先调用 `sendHelpCard`；若平台未挂载该能力，则保留 `replyText(formatHelp())` fallback。
- [ ] 步骤 5：新增私有格式化辅助逻辑，例如 `_appendModelFooter(text, session)`，内部使用 `_resolveSessionModel(session)` 与 `_formatModel(model)`。
- [ ] 步骤 6：footer 与正文之间使用分隔线，格式为 `---` 加 `模型：<model>`；解析不到模型时显示 `模型：未指定`。
- [ ] 步骤 7：仅对普通 Agent 最终文本追加 footer，不改变进度卡片内容、不影响空文本回复。
- [ ] 步骤 8：将命令去重 key 从 `cmd:${messageId}:${name}` 调整为包含参数的稳定 key，例如 `cmd:${messageId}:${name}:${args.join(' ')}`。
- [ ] 步骤 9：补充测试确认卡片 API 被调用、fallback 可用、点击模型按钮等价执行 `/model <provider>/<id>` 后更新 `session.model`。
