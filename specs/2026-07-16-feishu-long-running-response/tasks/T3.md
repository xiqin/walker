# Task 3: Driver 断流恢复与 completed 游标

## 描述

重构 OpenCode HTTP/SSE prompt 生命周期，在提交后断流时轮询恢复最终 completed assistant message，并修正 watcher 游标语义。

## 依赖

- 前置 task：T1、T2
- 外部依赖：OpenCode `/event`、`/prompt_async`、`/session/:id/message` 现有接口

## 涉及文件

| 操作 | 文件路径 | 说明 |
| ---- | -------- | ---- |
| 修改 | `src/drivers/agent-driver.js` | 补充 `prompt(..., options)` 与 `options.signal` 接口注释。 |
| 修改 | `src/drivers/opencode-driver.js` | 拆分 SSE open、prompt request、SSE idle，增加 submitted 后恢复 polling 和取消处理。 |
| 修改 | `src/drivers/opencode-session-watcher.js` | 将游标限定为已投递 completed assistant message，支持 pending 原地完成。 |
| 修改 | `test/opencode-driver.test.js` | 覆盖提交超时、断流恢复、取消、去重和失败游标。 |
| 创建 | `test/opencode-session-watcher.test.js` | 单测 watcher 的 completed 游标与 pending 原地完成。 |

## 复杂度

high，原因：需要协调专用 prompt SSE、长期 watcher、message polling 与取消信号，且必须避免丢失和重复投递。

## 实现步骤

- [ ] Driver 构造函数接收 `sseOpenTimeoutMs`、`promptRequestTimeoutMs`、`sseIdleTimeoutMs`、`messagePollIntervalMs`，所有允许关闭的值使用 `??` 保留 `0`。
- [ ] `prompt()` 组合调用方 signal 与内部 SSE controller；open timeout 为 `0` 时不创建 open deadline，为正数时抛出 `SSE_OPEN_TIMEOUT`。
- [ ] `/prompt_async` 请求传入独立 `timeoutMs` 与 signal；超时映射为 `PROMPT_REQUEST_TIMEOUT`，提交成功前失败不得进入恢复。
- [ ] 在提交前保存 watcher 的最后 delivered completed baseline；SSE 正常观察到 terminal event 时映射并返回事件。
- [ ] 提交成功后 SSE idle、断开或未带 terminal 正常结束时进入 recovering，保持长期 watcher 暂停，按配置间隔查询 message 列表。
- [ ] 恢复算法只选择 baseline 之后的 completed assistant message；若目标 message 仍 pending，不推进游标并继续轮询；完成后映射 text 与 done 事件，只返回一次。
- [ ] `finally` 仅在 prompt 成功得到 completed 结果并完成投递基线确认时推进 watcher 游标；transport 失败或取消绝不推进到 pending/未投递 message。
- [ ] watcher 初始化和 polling 只记录 completed assistant message；同一 ID 从 pending 变为 completed 时仍在后续 poll 中可见并投递一次。
- [ ] signal abort 时停止 SSE 与恢复 polling并抛 `ABORT_ERR`，随后安全 resume watcher；不把取消包装成业务错误。
