# Task 6: Dispatcher 取消集成与端到端验证

## 描述

让 Dispatcher 成为整轮唯一硬截止的拥有者，将 AbortSignal 传给 transport，并通过集成测试验证长任务恢复、取消抑制和幂等投递。

## 依赖

- 前置 task：T3、T4、T5
- 外部依赖：无；测试使用现有 fake Driver、Bridge 与 Feishu API

## 涉及文件

| 操作 | 文件路径 | 说明 |
| ---- | -------- | ---- |
| 修改 | `src/dispatch/message-dispatcher.js` | 每轮创建 AbortController，截止/取消时 abort，再调用 driver cancel/stop，并区分错误语义。 |
| 修改 | `test/message-dispatcher.test.js` | 覆盖 signal 透传、用户取消、deadline、transport 恢复状态和迟到输出抑制。 |
| 修改 | `test/integration-feishu-tui-sync.test.js` | 覆盖 v3 长任务、v2 兼容、迟到 final 和只投递一次。 |

## 复杂度

high，原因：这是最终跨模块集成点，需要保持 session 状态、进度卡片、watcher 去重和取消优先级一致。

## 实现步骤

- [ ] `_startTurnState()` 为每轮创建 `AbortController` 并保存在 turn state；`_enqueuePrompt()` 调用 `driver.prompt(..., { model, signal })`。
- [ ] 用户 `/cancel`、`/stop`、session 终止和 `WALKER_MAX_TURN_TIME_MINS` 到期时先 abort transport 等待，再调用现有 driver cancel/stop；所有路径幂等清理 deadline timer 与 heartbeat。
- [ ] deadline abort 使用 `TURN_DEADLINE_EXCEEDED` 或明确 reason，用户取消使用 cancel reason；transport 的 recovering 状态不将 session 标为 error，也不发送失败卡片。
- [ ] prompt 最终成功后沿现有 `_renderEvents()` 与 delivered-text 去重路径投递；watcher 或 tombstone 同时观察到相同 final 时只向飞书发送一次。
- [ ] 取消后保持 `cancelledTurnSessions` 抑制规则，确保 HTTP/SSE 恢复结果与 TUI 迟到 final 都不能修改已取消轮次。
- [ ] 集成 v3 accepted/heartbeat/final，使用短 lease/heartbeat 阈值证明执行超过旧固定阈值仍完成。
- [ ] 集成 v2 final 兼容与 transport_lost tombstone 补投，验证重复 final 返回成功但不重复发飞书消息。
- [ ] 运行全部定向测试和 `npm test`，修正仅由接口整合暴露的问题，不扩大功能范围。
