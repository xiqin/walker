# Task 5: 内置 TUI plugin 心跳协议

## 描述

升级内置 OpenCode TUI plugin 到 Bridge 协议 v3，在领取、执行和完成阶段分别上报 accepted、heartbeat 和 final。

## 依赖

- 前置 task：T1
- 外部依赖：OpenCode plugin API 与 Bridge `/register`、`/poll`、`/events` 现有 HTTP endpoints

## 涉及文件

| 操作 | 文件路径 | 说明 |
| ---- | -------- | ---- |
| 修改 | `src/opencode-hook/plugin-template.js` | 生成 v3 plugin，管理 delivery heartbeat 生命周期并保留 clear 逻辑。 |
| 创建 | `test/opencode-plugin-template.test.js` | 验证生成源码中的协议版本、状态上报、间隔配置和清理路径。 |

## 复杂度

medium，原因：模板为生成代码，需要在字符串源码中正确管理异步 timer，且不能干扰既有 session idle/final 聚合。

## 实现步骤

- [ ] 将生成物标记版本升级并在 register payload 中发送 `bridgeProtocolVersion: 3`。
- [ ] `getPluginSource()` 接收 heartbeat interval 参数并将安全整数内嵌到生成源码，不从运行时环境隐式读取。
- [ ] plugin poll 到 prompt delivery 后先上报 `deliveryState: 'accepted'`，成功后再调用 `session.promptAsync`。
- [ ] 为每个 active delivery 启动独立 heartbeat timer，按配置周期上报 `deliveryState: 'heartbeat'`；heartbeat 失败只记录并允许下个周期重试，不提前伪造 final。
- [ ] session idle 聚合到最终事件或执行报错时停止 heartbeat，并以 `deliveryState: 'final'` 上报现有 `events` 或 `error`。
- [ ] 所有 final、错误、session 切换和 plugin 清理路径都清除 timer；重复 OpenCode status/event 不得创建多个 heartbeat timer。
- [ ] 保持 clear delivery 的 control/result 格式和 bridge protocol v2 已有能力不变。
